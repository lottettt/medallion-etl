# Medallion ETL вЂ” Synthetic Medical Records

A **Bronze в†’ Silver в†’ Gold** data pipeline built with Apache Airflow and PostgreSQL, processing synthetic healthcare data to answer two clinical and operational business questions.

**Dataset source:** [Synthetic AL Medical Records вЂ” Kaggle](https://www.kaggle.com/datasets/mexwell/synthetic-al-medical-records)

---

The dataset contains 18 CSV files of realistic (but fully synthetic) patient records generated by Synthea. This pipeline ingests those files, cleans and validates them, then produces a star-schema analytical layer that answers:

### Q1 вЂ” Revenue Cycle
> Which payers and encounter types drive the highest outstanding balances, and what is the average days-to-payment?

Covers: encounters, payers, claims, claims transactions.

### Q2 вЂ” Care Quality
> Do patients with chronic conditions and active care plans have fewer emergency or inpatient revisits (readmissions within 30 days)?

Covers: patients, encounters, conditions, care plans.

## Architecture

```
dataset/ (CSV files)
    в”‚
    в–ј
hospital_op        в†ђ raw operational tables (COPY FROM CSV)
    в”‚
    в–ј bronze_*.py
dw_bronze          в†ђ append-only, MD5 row-hash dedup, full lineage
    в”‚
    в–ј silver_*.py
dw_silver          в†ђ cleaned, standardised, PDPA de-identified, DQ-flagged
    в”‚
    в–ј gold_*.py
dw_gold            в†ђ star schema, surrogate keys, pre-calculated KPIs
```

| Layer | Strategy | Key features |
|---|---|---|
| Bronze | Append-only INSERT | `_ingested_at`, `_source_file`, `_row_hash` вЂ” never overwrites source |
| Silver | TRUNCATE + full refresh | Dedup via `ROW_NUMBER()`, `_dq_status` (PASS/WARN), age group bucketing, PDPA de-identification |
| Gold | TRUNCATE + full refresh | SERIAL surrogate keys, `is_readmission` flag, `has_active_careplan` flag, `condition_category` (Chronic/Acute) |

**Data Quality** вЂ” every silver run logs per-column metrics (completeness, validity, accuracy) to `metadata.dq_log_results`. Each metric has a threshold; failures are marked `FAILED` and visible in that table.

## Setup

### Prerequisites
- Docker & Docker Compose
- The dataset CSV files from Kaggle placed in `dataset/`

### 1. Download the dataset

Go to [https://www.kaggle.com/datasets/mexwell/synthetic-al-medical-records](https://www.kaggle.com/datasets/mexwell/synthetic-al-medical-records), download and extract all CSV files into the `dataset/` folder.

```
dataset/
в”њв”Ђв”Ђ patients.csv
в”њв”Ђв”Ђ encounters.csv
в”њв”Ђв”Ђ claims.csv
в”њв”Ђв”Ђ claims_transactions.csv
в”њв”Ђв”Ђ conditions.csv
в”њв”Ђв”Ђ careplans.csv
в”њв”Ђв”Ђ payers.csv
в””в”Ђв”Ђ ... (18 files total)
```

### 2. Start everything

```bash
docker-compose up -d
```

On first run this automatically:
1. Starts PostgreSQL
2. Migrates the Airflow metadata database and creates the admin user
3. Runs `ingest.py` вЂ” creates the `hospital_op` schema and bulk-loads all CSVs
4. Runs `init_db.py` вЂ” creates `dw_bronze`, `dw_silver`, `dw_gold`, and `metadata` schemas
5. Starts the Airflow webserver and scheduler

On subsequent runs `ingest.py` and `init_db.py` detect that data already exists and skip вЂ” no duplicates, no errors.

### 3. Open Airflow

Go to [http://localhost:8080](http://localhost:8080) and log in with:

```
Username: admin
Password: admin
```

### 4. Run the pipelines

Trigger either DAG from the Airflow UI:

| DAG | Business question |
|---|---|
| `q1_revenue_cycle` | Revenue Cycle analysis |
| `q2_care_quality` | Care Quality analysis |

Each DAG runs the full bronze в†’ silver в†’ gold pipeline for its question. They can be triggered independently or run on the default `@daily` schedule.

### 5. Tear down

```bash
docker-compose down          # stop containers, keep data volume
docker-compose down -v       # stop containers AND delete all data
```

## Database schemas

| Schema | Purpose |
|---|---|
| `hospital_op` | Raw operational tables вЂ” direct CSV dump, never modified |
| `dw_bronze` | Append-only bronze layer with lineage columns |
| `dw_silver` | Cleaned and validated silver layer with DQ flags |
| `dw_gold` | Star schema gold layer вЂ” ready for BI tools |
| `metadata` | DQ run logs (`metadata.dq_log_results`) |

### Querying DQ results

```sql
-- See all failed DQ checks from the latest run
SELECT run_id, table_name, column_name, metric_type, metric_value, threshold
FROM metadata.dq_log_results
WHERE status = 'FAILED'
ORDER BY checked_at DESC;
```

### Q1 example query

```sql
-- Average days-to-payment and outstanding balance by payer and encounter class
SELECT
    dp.name              AS payer,
    fc.encounter_class,
    ROUND(AVG(fc.avg_days_to_payment), 1) AS avg_days_to_payment,
    ROUND(SUM(fc.total_outstanding), 2)   AS total_outstanding
FROM dw_gold.fact_claims fc
JOIN dw_gold.dim_payers dp ON dp.payer_key = fc.payer_key
WHERE fc.claim_status = 'OPEN'
GROUP BY dp.name, fc.encounter_class
ORDER BY total_outstanding DESC;
```

### Q2 example query

```sql
-- Readmission rate for patients with vs without active care plans
SELECT
    has_active_careplan,
    COUNT(*)                                              AS total_encounters,
    SUM(is_readmission::INT)                              AS readmissions,
    ROUND(100.0 * SUM(is_readmission::INT) / COUNT(*), 2) AS readmission_rate_pct
FROM dw_gold.fact_encounters
WHERE encounter_class IN ('inpatient', 'emergency')
GROUP BY has_active_careplan;
```
